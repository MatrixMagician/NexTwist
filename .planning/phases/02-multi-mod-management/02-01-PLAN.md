<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB migration ↔ existing Phase-1 user data | V2 runs automatically on Store::open over a live user DB; a wrong/destructive migration could corrupt or drop Phase-1 deployment state (manifest/journal/vanilla), breaking the pristine guarantee. |
| Persisted token columns ↔ in-memory enums | `plugin_state.kind` is a free-text token that could be corrupt/garbage on disk. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | V2 migration over Phase-1 DB | mitigate | V2 is strictly ADDITIVE (only CREATE TABLE/INDEX + one INSERT into the NEW profile table); it never ALTERs/DROPs/UPDATEs V1 tables (managed_game/deployed_file/op_journal/vanilla_backup). BLOCKING test asserts V1 tables + data survive and Default profile is created. |
| T-02-02 | Tampering | plugin_state.kind token decode | mitigate | Use the double-Result `StoreError::Corrupt` row mapper (manifest.rs pattern); a corrupt token surfaces an error, never a silent wrong value. Unit-tested. |
| T-02-03 | Denial of Service (self) | profile uniqueness | accept | `UNIQUE(appid, name)` rejects duplicate profile names per game; surfaced as StoreError::Db to the caller. Low risk, single-user desktop. |
| T-02-SC | Tampering | npm/cargo installs | mitigate | This plan adds NO new package (only a deny.toml allowance comment + MSRV bump). libloot install + legitimacy checkpoint is Plan 04 (T-04-SC). |
</threat_model>

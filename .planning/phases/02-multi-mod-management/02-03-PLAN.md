<objective>
Deliver the conflict vertical slice end-to-end (store → resolver → deploy → Tauri command → Svelte view → pristine test): a user with multiple enabled mods can SEE file-level conflicts, SET mod priority so a chosen mod deterministically wins, and DEPLOY the winner set — with the round-trip-pristine guarantee preserved.

Implements CONF-01 (file-level + per-mod conflict view), CONF-02 (ordered priority/rank winner — D-01), CONF-03 (deterministic winner deploy through the unchanged safe engine), D-02 (file-level + per-mod view), D-03 (winning (mod,file,hash) recorded in manifest — via the existing engine), and D-04 (priority changes pending until explicit Deploy). The conflict resolver is a pure fold; `engine::deploy` stays the unchanged disk primitive (NEVER bypassed).

After Task 1 a user can call resolve and see winners; after Task 2 the winner set deploys with one owner per path and survives a purge to pristine; after Task 3 the user does the whole thing from the UI with an explicit pending-vs-deployed banner and Deploy button.

Purpose: make the manifest meaningful with many mods — the user controls who wins, deterministically and reversibly.
Output: `deploy::conflict` module, the StagedFiles multi-root contract decision, conflict Tauri commands, the Conflict view, and a pristine redeploy integration test.
</objective>

    - resolve over two mods that share Data/x.esp: the lower-rank (higher-priority) mod is the winner; output has exactly ONE entry for Data/x.esp; FileConflict.providers lists both mod ids, winner = the lower-rank mod.
    - resolve where no two mods share a path: zero FileConflicts; every staged file appears once.
    - changing the winning mod's rank above the other flips the winner (CONF-02).
    - resolve NEVER emits two entries for the same target_rel (Pitfall 3 — assert no duplicate target_rel in output).
    - a winner path that lexically escapes its mod's staging root is rejected (DeployError::PathEscape) — reuse guard semantics.

  <action>

**Contract decision (flag in SUMMARY):** today `StagedFiles { staging_root, files }` has ONE staging_root, but multi-mod winners come from DIFFERENT roots. Choose ONE and document it:
- Option A (preferred): add a sibling type the deploy path accepts, e.g. `pub struct WinnerFile { pub staging_root: PathBuf, pub rel: PathBuf }` and a `deploy_winners(store, game, &[WinnerFile])` that reuses `deploy_inner` per-file logic; keep `StagedFiles`/`deploy` unchanged for Phase-1 callers.
- Option B: extend `StagedFiles` to hold `Vec<(PathBuf, PathBuf)>`.
Pick A unless reading engine.rs shows B is materially simpler; keep `engine::deploy`'s existing single-root behavior intact either way (Phase-1 tests must still pass).

Implement:
1. `pub struct ModInput { pub mod_id: i64, pub staging_root: PathBuf, pub rank: u32 }` (doc: lower rank = higher priority, 1-based).
2. `pub fn resolve(mods: &[ModInput]) -> Result<(<winner set type>, Vec<FileConflict>), DeployError>`: walk each enabled mod's staged tree with walkdir → build `BTreeMap<target_rel, Vec<(rank, mod_id, staging_root)>>`; for each path sort providers by rank ascending, winner = providers[0]; emit the winner's (staging_root, rel); produce a `FileConflict { target_rel, providers: all mod_ids, winner: winner mod_id }` ONLY where providers.len() > 1 (drives CONF-01). Use deterministic ORDER (BTreeMap) so output is stable. For each winner path, assert lexical containment within its staging root (reuse/duplicate `guard_within_root`/`lexical_normalize`; promote to a shared helper if cleanest — Security §V5/V12).
3. Wire the winner set into the chosen deploy path; the existing engine records the winning (mod,file,hash) into the manifest (D-03) — winner FileEntry.source_mod = winner mod_id, hash via the existing blake3 path.
4. `crates/deploy/src/lib.rs`: `pub mod conflict;` and `pub use conflict::{resolve, ModInput, ...};`.
5. Unit tests in conflict.rs `#[cfg(test)]` covering all behaviors above (build staged trees with `testkit::fake_staged_mod`); the no-duplicate-target_rel assertion is mandatory.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mod-supplied staged paths ↔ deploy root | Conflict winner paths come from mods' staged trees; a crafted relative path could try to escape the staging root / game Data root. |
| Webview (untrusted UI input) ↔ Tauri commands | mod_id/rank/appid cross the IPC boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Tampering / EoP | conflict::resolve winner paths | mitigate | Each winner path asserted lexically contained within its mod's staging root (reuse guard_within_root/lexical_normalize); escapes → DeployError::PathEscape. Unit-tested. (Staged trees were already zip-slip/symlink-validated in Phase-1 extract — defence in depth.) |
| T-02-07 | DoS (self) | two owners for one path | mitigate | Resolver dedups to a single winner per target_rel BEFORE deploy; no-duplicate-path unit test; the existing engine's crash-recovery already handles a partial op. |
| T-02-08 | Tampering (integrity) | multi-mod deploy then purge | mitigate | conflict_redeploy.rs asserts byte-for-byte pristine after deploy+purge AND after a rank-change redeploy (testkit DIR_SENTINEL harness). |
| T-02-09 | Spoofing | IPC command args | accept | Single-user desktop; adapters validate appid via require_game; mod_id/rank are bounded ints the store treats as data. |
</threat_model>

# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2 (Multi-Mod Management). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase extends the Phase-1 "functional-minimal Svelte 5" UI (`frontend/src/routes/+page.svelte`). It introduces four new dense, interactive surfaces — Conflict view, Plugin manager, LOOT sort, Profiles — without adding a component library. Visual tokens below are extracted from the existing Phase-1 stylesheet so the new surfaces are visually continuous with what already ships.

**Design priority for this phase:** clarity and SAFETY over polish. Every disk-mutating action (Deploy, profile switch) is explicit and confirmation-gated; "who wins" and "pending vs deployed" must be unambiguous at a glance.

## Spacing Scale

Phase-1 used rem-based ad-hoc spacing (0.2/0.3/0.4/0.6/0.8/1/1.25/1.5rem). For Phase 2 these are normalized to a 4px grid (1rem = 16px) so the new dense tables/lists are consistent. Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, reorder-button padding, glyph-to-text gap, table cell vertical padding |
| sm | 8px | Compact list-row gaps, toggle-to-label gap, button vertical padding |
| md | 16px | Default element spacing, section inner padding, table cell horizontal padding |
| lg | 24px | Section padding, gap between the four surfaces, modal padding |
| xl | 32px | Layout gaps between major column blocks (e.g. mod list / detail panel) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (reserved; unlikely in this single-page layout) |

Exceptions:
- **Reorder/drag handle and toggle hit targets: minimum 32px square** (desktop pointer target; this is a desktop app, not touch — 44px touch target not required). Rows in the priority list and plugin list must be at least 32px tall so reorder controls are comfortably clickable.
- Dense data tables (conflict file list) may use **4px vertical / 8px horizontal** cell padding to fit more rows legibly on WebKitGTK; this is below `md` intentionally for density and is the only sanctioned sub-`md` content padding.

## Typography

Sizes carried from Phase-1 (`h1` 1.5rem, `h2` 1.15rem→normalized to 18px, body 1rem, `.paths` 0.85rem→14px) and capped at 4 roles. 16px base, two weights only (400 regular, 600 semibold). Phase-1 base line-height was 1.4; body raised to 1.5 for the denser reading content, headings 1.2.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (page `h1`, surface section `h2`) | 24px | 600 | 1.2 |
| Heading (sub-section `h3`/`h4`, panel titles) | 18px | 600 | 1.2 |
| Body (list rows, table cells, descriptions, button labels) | 16px | 400 | 1.5 |
| Label / meta (paths, hashes, mod-count badges, secondary captions) | 14px | 400 (600 only for emphasis like a mod name) | 1.5 |

Monospace: file paths, `target_rel`, hashes, and `plugins.txt` line previews render in the browser default monospace with the Phase-1 `code` treatment (light-gray background, 3px radius, `word-break: break-all`). Monospace is a treatment of the Label/Body roles, not a fifth size.

## Color

Light theme continues from Phase-1 (no dark mode in scope). 60/30/10 split mapped onto the existing palette; the amber/green/red semantic colors are preserved verbatim so warnings read identically across phases.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Page/main background, the large data-table and list area |
| Secondary (30%) | `#f3f3f3` (surfaces) with `#ccc` borders, section radius 8px | Section/card chrome, table header row, `code` chips, modal backdrop panel — the bordered-section container Phase-1 already uses |
| Accent (10%) | `#0a66c2` (calm blue) | Reserved — see list below |
| Destructive | `#cf222e` (Phase-1 `.err` red) | Destructive/disk-mutating action emphasis only |

Accent reserved for (explicit — never "all interactive elements"):
- The single **primary Deploy CTA** when changes are pending (the one action that commits disk changes for conflicts/priority).
- The **active profile** indicator in the profile selector (which profile is currently deployed).
- The **selected row** highlight in the priority list / plugin list (subtle left-border or background tint at ~12% accent).
- Plain secondary buttons (Sort with LOOT, Add profile, reorder ▲▼) stay neutral (Phase-1 default button chrome), NOT accent.

Additional semantic colors (carried verbatim from Phase-1 — these are status, not part of the 10% accent budget):

| Semantic | Value | Usage |
|----------|-------|-------|
| Success | `#1a7f37` | "Deployed / up to date", "Pristine", winner confirmation |
| Warning | text `#9a6700`, bg `#fff8e5`, border `#e6c200` | LOOT critical warnings (dirty plugins, missing masters), masters-first violations, **pending/stale-deploy banner**, cross-device fs warnings |
| Muted | `#777` | Empty-state text, "overwritten-by" loser rows, disabled-plugin rows |

## Surface & Interaction Contracts

These are the load-bearing contracts for this phase — the data is dense and the safety semantics must be visually unambiguous. Executor treats these as the source of truth.

### A. Conflict View (CONF-01/02/03)

1. **Mod priority list** — an ordered, reorderable list; **top = highest priority = wins**. Each row: drag handle / `▲▼` reorder buttons (xs gap), mod name (Body 600), enabled toggle, and a small count of files it wins / loses (Label 14px, muted). Order is 1-based and shown. Reorder via buttons is the baseline; native HTML5 drag is a progressive enhancement, never the only path (keyboard/click reorder must work on WebKitGTK).
2. **File-level conflict table** (CONF-01) — columns: `target_rel` (monospace, `break-all`), **Provided by** (list of mods), **Winner** (the highest-priority provider, marked with a filled `●` accent dot + mod name in 600). Losers shown muted with a strikethrough-free "overwritten" label. Table header row uses Secondary `#f3f3f3`.
3. **Per-mod overwrite summary** (CONF-01) — for the selected mod: two short lists, "Overwrites" and "Overwritten by", each a count + expandable mod/file list.
4. **Pending vs deployed indicator (critical)** — when the chosen winner/priority set differs from the deployed set, show a persistent **Warning-styled banner** at the top of the surface: heading "Changes pending" + body "Priority changes aren't on disk yet. Deploy to apply." The **Deploy** button takes the Accent color in this state; when in sync it is neutral/disabled with caption "Up to date".

### B. Plugin Manager (PLUGIN-01/02)

1. **Plugin list** — reorderable rows of `.esp/.esm/.esl`; each row: `▲▼` reorder, enable checkbox/toggle, plugin filename (monospace Body), and a type badge (`ESM`/`ESL`/`ESP`, Label 14px). Disabled plugins render muted.
2. **Masters-first invariant (hard, visually evident)** — masters (`.esm` + ESL-flagged) are grouped/pinned above regular `.esp` with a visible group divider labeled "Masters (load first)". Reorder attempts that would violate masters-first are **prevented** (control disabled) and, if attempted, surface a Warning-styled inline message: "Masters must load before regular plugins." This is non-negotiable structural ordering, not a preference.
3. **plugins.txt preview** — an optional read-only monospace preview of the asterisk-enabled output (`*Enabled.esp` / disabled without `*`) so the user can see exactly what will be written to the prefix. Writing to `plugins.txt` is part of profile/plugin apply, not a separate surprise action.

### C. LOOT Auto-Sort (PLUGIN-03)

1. **"Sort with LOOT"** — a single neutral (non-accent) button. On click it **proposes** a new order shown as a reviewable diff against the current order (moved plugins highlighted); nothing is applied silently.
2. **Review/Apply flow** — two buttons after a proposal: **Apply sorted order** (writes the proposed order into the editable plugin list; still hand-editable afterward) and **Discard**. No silent auto-apply.
3. **Critical warnings** — dirty plugins and missing masters render in the Warning style as a list above the proposal. v1 only surfaces these; there is no cleaning UI.

### D. Profiles (PROF-01/02/03)

1. **Profile selector** — a per-game dropdown/segmented list at the top of the game's panel. The **active (deployed) profile** is marked with the Accent indicator + "active" Label. Actions: Create, Rename, Delete (Delete is destructive → see copywriting).
2. **Switching is confirmation-gated (disk-mutating)** — selecting a different profile does NOT immediately mutate disk. It opens a **confirmation modal** explaining: "Switching purges the current deployment and deploys '<name>'. The game stays safely reversible." Confirm button is Accent; Cancel is neutral. Only on confirm does the safe engine run purge-old → deploy-new.
3. **Per-profile preservation** — the UI makes clear each profile keeps its own enabled-mod set, priority order, and plugin order; switching visibly reloads those lists.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA (conflicts/priority) | "Deploy" (when pending) / "Up to date" (disabled, in-sync) |
| Primary CTA (plugins) | "Save plugin order" |
| Primary CTA (LOOT) | "Sort with LOOT" → then "Apply sorted order" / "Discard" |
| Primary CTA (profiles) | "Switch profile" / "Create profile" |
| Empty state — conflicts | Heading: "No conflicts" · Body: "No two enabled mods write the same file. Enable more mods or adjust priority to see conflicts here." |
| Empty state — plugins | Heading: "No plugins found" · Body: "No .esp/.esm/.esl files in the enabled mods or game Data folder. Install or enable a mod that adds plugins." |
| Empty state — profiles | Heading: "Default profile" · Body: "This game has one profile. Create another to keep separate mod/plugin setups." |
| Pending/stale banner | Heading: "Changes pending" · Body: "Priority changes aren't on disk yet. Deploy to apply." |
| Error state (generic, Phase-1 pattern) | "{action} failed: {reason}" — keep the Phase-1 `run()` `label + error` convention; surface the backend error string verbatim so safety failures are never hidden. |
| Error — plugins.txt write | "Couldn't write plugins.txt: {reason}. Your load order wasn't changed in-game. Check the game's Proton prefix is resolved." |
| Error — LOOT unavailable | "LOOT sort is unavailable: {reason}. You can still set load order by hand." |
| Destructive confirmation — Profile switch | "Switch to '{name}'?": "This purges the current deployment and deploys '{name}'. Your game stays fully reversible. Continue?" — Confirm: "Switch", Cancel: "Cancel". |
| Destructive confirmation — Delete profile | "Delete '{name}'?": "This removes the profile and its mod/plugin selections. Staged mod files are kept. This can't be undone." — Confirm: "Delete" (Destructive red), Cancel: "Cancel". |
| Destructive confirmation — Purge (carried from Phase-1) | "Purge all mods?": "Restores the game folder to pristine vanilla. Mods stay staged and can be redeployed." — Confirm: "Purge", Cancel: "Cancel". |

Destructive-action affordance rule: every disk-mutating action (Deploy after profile switch, profile Switch, Delete profile, Purge) requires an explicit confirmation step. Delete uses Destructive red on its confirm button; Deploy/Switch use Accent (committing intended state, not deleting data).

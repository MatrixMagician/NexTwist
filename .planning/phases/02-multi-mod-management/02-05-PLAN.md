<objective>
Deliver the profile vertical slice end-to-end (store → reconcile → purge/deploy/plugins.txt → Tauri command → Svelte view → pristine switch test): a user can create multiple independent per-game profiles, switch the active profile to change which mods/plugins/order are deployed, with each profile preserving its own state, and the round-trip-pristine guarantee holding across switches.

Implements PROF-01 (create multiple profiles — D-16 store already supports it from Plan 01), PROF-02 (switch reconciles through the safe engine — D-15: purge(old) → deploy(new winner set) → apply new plugins.txt, confirmation-gated), PROF-03 (each profile preserves its own enabled-mod set + priority + plugin order — D-13/D-14 shared staging store, lightweight references). Reuses the Plan-03 conflict winner deploy path and the Plan-04 `apply_load_order`. The engine is NEVER bypassed; switching is a full purge-to-pristine then deploy (Pitfall 4 — no diff-deploy in v1).

After Task 1 a user can switch profiles headlessly with pristine guaranteed. After Task 2 they do it from a confirmation-gated UI.

Purpose: independent, instantly-switchable mod setups per game — the capstone of multi-mod management — without ever compromising reversibility.
Output: `deploy::profile` reconcile module, the profile-switch pristine test, profile Tauri commands, and the Profile selector + confirmation modal.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Profile switch ↔ on-disk game state | A switch purges then deploys; a crash mid-switch must still leave the game safely reversible (the core safety guarantee). |
| Webview ↔ profile Tauri commands | profile_id / name / appid cross IPC; a destructive switch/delete is triggered from the UI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-14 | Tampering (integrity) | profile switch interrupted | mitigate | switch = purge(old) → deploy(new) through the EXISTING journaled, crash-safe engine; `recover_on_launch` replays a partial op on next start; profile_switch.rs regression-locks byte-for-byte pristine across A→B→A switches (Pitfall 4). |
| T-02-15 | Tampering | stale files leak between profiles | mitigate | Full purge-to-pristine BETWEEN profiles (never diff-deploy in v1) so profile A's unique files cannot survive into profile B; asserted by the cross-switch pristine test. |
| T-02-16 | Repudiation / accidental loss | profile switch/delete from UI | mitigate | Every disk-mutating profile action is confirmation-gated (UI-SPEC §D.2 + Copywriting); delete keeps staged mod files (only references removed); set_active only after a successful deploy. |
| T-02-17 | DoS (self) | duplicate/invalid profile | accept | store UNIQUE(appid,name) rejects dupes; single-user desktop, low risk. |
</threat_model>

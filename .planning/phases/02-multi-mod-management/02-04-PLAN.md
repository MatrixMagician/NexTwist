<objective>
Deliver the plugin + LOOT vertical slice end-to-end (scan → libloot → masterlist → plugins.txt → Tauri command → Svelte view): a user can see the plugins from their enabled mods + game Data/, enable/disable them, adjust load order written correctly to plugins.txt in the Proton prefix, and auto-sort via LOOT with a propose-then-apply flow.

Implements PLUGIN-01 (enable/disable — D-06 discovery scans staged trees + Data/), PLUGIN-02 (load order written as asterisk plugins.txt at the prefix AppData location, masters-first — D-05/D-07/D-08), PLUGIN-03 (LOOT auto-sort — D-09/D-10/D-11/D-12: libloot sort, masterlist cached, sort+critical-warnings scope, propose-review-apply, no silent apply). Built on the Plan-02 libloot wrapper (use its verified API surface) and Plan-01 plugin_state store. plugins.txt is DERIVED from DB plugin_state (regenerable; not the pristine-invariant) per RESEARCH Open Question 3.

After Task 1 a user can list plugins with type badges. After Task 2 they can enable/order and write plugins.txt at the prefix. After Task 3 they can LOOT-sort with review. NEVER hand-roll plugins.txt or the sort (libloot owns both).

Purpose: plugins load correctly in-game under Proton — the in-game-observable half of multi-mod management.
Output: scan + loot + masterlist modules, plugin Tauri commands, the Plugin manager view, and headless plugin tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| NexTwist ↔ raw.githubusercontent.com (masterlist) | masterlist.yaml fetched over the network; MITM/tampering risk if not TLS-pinned to the right host/branch. |
| Mod-supplied plugin filenames ↔ plugins.txt write | Plugin names come from staged mod trees; an oversized/crafted name could try to escape the prefix when writing plugins.txt. |
| Webview ↔ Tauri plugin commands | plugin name / order / enabled cross IPC. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-10 | Tampering | masterlist fetch | mitigate | reqwest rustls-tls (no OpenSSL) over HTTPS to a pinned host `raw.githubusercontent.com` + pinned repo `loot/<game>` + pinned branch `v0.29`; bundled CC0 snapshot fallback. Masterlist is parsed by libloot (trusted CC0 source). |
| T-02-11 | Tampering / EoP | plugins.txt write path | mitigate | plugins.txt path is libloot's `active_plugins_file_path` derived strictly from the resolved prefix (`appdata_local_path` bounded under the prefix root); the write stays inside the resolved Proton prefix. Plan-02 spike already asserts the bounded path. |
| T-02-12 | Tampering | malicious/oversized plugin names | mitigate | scan collects only files actually present in validated staged trees / Data/ (Phase-1 extract already rejected zip-slip/symlink); libloot validates plugin headers; plugin names are treated as opaque filenames, never as paths to join outside the prefix. |
| T-02-13 | Tampering (integrity) | plugins.txt vs pristine invariant | accept | plugins.txt lives in the Proton prefix, NOT the game install Data/ — it is NOT part of the byte-for-byte-pristine game-folder invariant and is regenerable from DB plugin_state (RESEARCH OQ3). Crash mid-write is recoverable by re-deriving from the DB. |
| T-02-SC | Tampering | esplugin/reqwest installs | mitigate | esplugin (if added) is a transitive libloot dep, same LOOT author, GPL-3.0 already allowed (covered by the Plan-02 libloot checkpoint family). reqwest is the project-sanctioned HTTP client (CLAUDE.md). No new SLOP packages. |
</threat_model>

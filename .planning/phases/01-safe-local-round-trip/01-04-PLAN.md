<objective>
Build the crown jewel: `crates/deploy` — the reversible deployment engine. This plan delivers the complete safe round-trip spine: a per-target filesystem-capability probe, a `DeploymentMethod` trait ladder (reflink -> hardlink -> symlink -> copy) chosen per-target at deploy time with EXDEV/CrossesDevices fallback, the intent-before-act operation-journal protocol with idempotent replay/recovery on launch, backup-before-overwrite into a content-addressed vanilla store, and the `deploy()` / `purge()` / `recover_on_launch()` orchestration — proven by the centerpiece crash-recovery test and the proptest round-trip-pristine assertion on tmpfs and the dev btrfs filesystem.

Purpose: This is the differentiating, irreplaceable value of the entire product (PROJECT.md Core Value). It completes the Walking Skeleton spine: detect (Plan 02) -> install to staging (Plan 03) -> DEPLOY non-destructively -> PURGE to byte-for-byte pristine, surviving a crash mid-deploy. Crash-safety here cannot come from SQLite WAL alone (a `link()` syscall and its DB row are two operations that cannot be made atomic), so the journaling/idempotency protocol gluing fs syscalls to the DB is the one thing that must be hand-written and exhaustively tested.

Output: `crates/deploy` with probe, method ladder, journal, backup, and engine modules, plus five integration/unit tests: `fs_probe`, `method_ladder` (forces EXDEV via a second temp fs), `vanilla_restore`, `round_trip_pristine` (proptest), and `crash_recovery` (the centerpiece). Casefold normalization (DEPLOY-08) and verify/repair drift (DEPLOY-07) are added in Plan 05 on top of this engine.
</objective>

  <behavior>
    - deploy records a `pending` journal intent BEFORE each file op, backs up a pre-existing vanilla file, performs the idempotent file op, then flips the intent to `done` and writes the manifest row in one tx.
    - purge reads ONLY the manifest (never a directory scan): removes each recorded link, restores each backed-up vanilla original, and asserts the result; orphans are reported, not blindly deleted.
    - deploying a mod that REPLACES a vanilla file backs up the original to the content-addressed store; purge restores the exact original bytes (vanilla_restore).
    - recover_on_launch replays every non-`done` journal row idempotently (finish a deploy / undo a purge) and ends in a consistent state.
    - A pure-add mod (no overwrites) deploys and purges with no vanilla backup rows.
  </behavior>
  <action>
    `journal.rs`: implement the intent-before-act protocol over the store primitives — `begin_op` writes a durable `pending` row (committed before the syscall), `mark_done` flips it after the manifest row is written, `replay(store, game)` iterates `pending_ops()` and rolls each forward (re-apply idempotent deploy) or back (idempotent remove + restore vanilla) by `kind`. The intent-commit uses the FULL-synchronous path (set in store::open) for durability; re-applying a completed op is a no-op by construction.

    `backup.rs`: `backup_vanilla_if_absent(store, game, target)` — if the target does not exist OR was deployed by us (`is_ours` via manifest), do nothing; else blake3-hash the original, copy it into `<app_data>/originals/<appid>/<hash>` only if that content key is absent (content-addressed dedupe), and `record_vanilla(appid, target_rel, hash)`. `restore_vanilla(store, game, target)` copies the stored original back. This is the single most important safety mechanism (Pitfall 3) — corruption here is otherwise only fixable by Steam re-verify.

    `engine.rs`: `deploy(store, game, staged)` — for each staged file, resolve the target relpath under `game.install_dir/Data`, run `journal::begin_op(pending)`, `backup::backup_vanilla_if_absent`, choose+apply the method idempotently, then `journal::mark_done` + `store::record_deployed_file` in one tx. `purge(store, game)` — drive ENTIRELY from `list_deployed_files`: remove each recorded path idempotently, restore any recorded vanilla original, drop the manifest+vanilla rows, and run a final pristine check (snapshot Data/ vs recorded provenance) reporting orphans rather than scanning-and-deleting. `recover_on_launch(store, game)` calls `journal::replay`. NEVER write a manifest row before the syscall succeeds without a preceding pending intent; NEVER purge by directory scan.

    `tests/vanilla_restore.rs`: build a fake game tree with a vanilla `Data/x.dds`, deploy a staged mod that replaces `Data/x.dds`, assert the deployed bytes differ, purge, then assert (via testkit `assert_trees_identical`) that `Data/x.dds` is byte-for-byte the original.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| staged mod tree → game Data/ | Deployed (third-party-derived) files cross into the real game install; must never corrupt vanilla state |
| app crash → on-disk + DB state | An interrupted operation must always be recoverable to a consistent/pristine state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Denial of Service (to the user's game) | crash-induced vanilla-file loss | mitigate | Backup-before-overwrite into content-addressed vanilla store + intent-before-act journal + idempotent replay on launch (DEPLOY-04/06) |
| T-01-12 | Tampering | non-atomic manifest -> orphans -> purge non-pristine | mitigate | Operation-journal intent-before-act; purge driven only by manifest (never directory scan); idempotent purge; orphans reported not blindly deleted (Pitfall 4) |
| T-01-13 | Tampering | overwrite vanilla in place with no backup (unrecoverable) | mitigate | Always backup pre-existing non-ours files before overwrite; staged files read-only + reflink (independent inode) preferred over hardlink (Pitfall 3) |
| T-01-14 | Elevation | Steam update writes THROUGH a directory symlink into staging | mitigate | Deploy per-file only; never symlink whole directories into Data/ (Pitfall 2) |
| T-01-15 | Information Disclosure | deploy/purge writing outside the resolved game dir + app-data | mitigate | All targets resolved under `game.install_dir/Data`; vanilla store under app-data; no path escapes (V4 access control) |
</threat_model>

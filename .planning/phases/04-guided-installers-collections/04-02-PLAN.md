<objective>
Deliver the user-facing FOMOD install wizard end-to-end (FOMOD-01, FOMOD-02): installing a mod that contains `fomod/ModuleConfig.xml` opens a guided step-by-step wizard, the user makes option choices, conditions re-evaluate live, and pressing Install shows the dry-run conflict preview before NexTwist writes anything to staging. On accept, the resolved plan routes through the validated extract→staging path and the mod becomes an ordinary `ManagedMod`.

Purpose: This is the FOMOD-01 capability the user actually sees. It consumes Plan 01's headless `crates/fomod` resolver (no FOMOD logic in the adapter — the wizard re-evaluation calls the headless `condition`/`resolve`). The two safety gates from the UI-SPEC (dry-run conflict preview) are non-skippable.

Output: A thin `commands/fomod.rs` adapter (`parse_fomod`, `resolve_fomod`, `apply_fomod`); registration in `commands/mod.rs` + `lib.rs`; `api.ts` invoke wrappers + report interfaces; the FOMOD wizard surface in `+page.svelte` per UI-SPEC §A; a `fomod` dependency added to `src-tauri/Cargo.toml`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webview → fomod command adapter | The IPC carries the archive path + the user's selection JSON into the shell |
| FOMOD plan → staging write | apply_fomod is the only path that turns a resolved plan into disk writes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-05 | Tampering | apply_fomod writing files outside Data/ via a crafted destination | mitigate | apply routes every file through `extract::install_archive` (zip-slip/symlink/`..` defense unchanged) + the deploy `conflict::resolve` path-escape guard; the adapter adds no new write primitive |
| T-04-06 | Elevation of Privilege | business logic leaking into the Tauri adapter (bypassing tested guards) | mitigate | Anti-Pattern-4: the adapter only locks AppState + calls headless `fomod`/`extract`/`store`; grep-verified no FOMOD logic inline |
| T-04-07 | Information Disclosure | a half-broken wizard mis-installing a malformed FOMOD silently | accept→mitigate | malformed FOMOD returns the verbatim `FomodError` and offers the plain-mod fallback (UI-SPEC §A.8) — no silent install path exists |
| T-04-SC | Tampering | npm/cargo package installs in this plan | mitigate | No NEW package added (the `fomod` workspace crate + `quick-xml` were audited in Plan 01); `cargo deny` runs at wave merge |
</threat_model>

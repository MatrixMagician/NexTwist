# Phase 4 — UI Design Contract

> Visual and interaction contract for Phase 4 (Guided Installers & Collections). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase extends the existing "functional-minimal Svelte 5" UI (`frontend/src/routes/+page.svelte`) established in Phases 1–3. It adds three new surfaces — (A) a **FOMOD install wizard** (step-by-step, radio/checkbox groups, image+description per option, live conditional re-evaluation, dry-run conflict preview), (B) a **Collections** view (browse/select, a resolve report shown *before* any disk write, bulk per-mod + overall download progress, a Premium-required notice), and (C) **install/deploy/uninstall affordances for an installed Collection** (reusing the Phase-2 profile patterns). No component library is added; all visual tokens below are inherited verbatim from the live Phase-1/2/3 stylesheet so the new surfaces are visually continuous with what already ships.

**Design priority for this phase:** legibility of *consequence before action*. Two safety gates must be visually unmissable: (1) the FOMOD **dry-run conflict preview** that resolves the full file-install plan and surfaces conflicts *before touching staging*, and (2) the Collection **resolve report** that lists every pinned mod with its archived/unavailable/off-Nexus status *before any download or disk write*. The user must always see what is missing or in conflict and explicitly accept it before NexTwist mutates anything. The Premium-only gate for Collections must read as a clear factual notice, not a paywall nag. Every download stream and live conditional re-evaluation is async/event-driven so the UI never freezes.

## Spacing Scale

Inherited unchanged from the Phase-1/2/3 contract (4px grid, `1rem = 16px`). The FOMOD wizard, the Collections views, and the installed-Collection affordances reuse the existing section / list / row / modal spacing so they sit flush with the prior surfaces.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, glyph-to-text gap, option-radio-to-label gap, progress-bar-to-label gap, dense table padding |
| sm | 8px | Compact list/option-row gaps, status-dot-to-text gap, button vertical padding, wizard option-group inner gaps, progress-bar height |
| md | 16px | Default element spacing, section inner padding, gap between wizard step body and nav bar, gap between resolve-report rows |
| lg | 24px | Section padding, gap between major blocks within a surface, modal padding (wizard runs inside the existing `.modal`/`.overlay` pattern) |
| xl | 32px | Layout gaps between major surfaces (Collections browse vs resolve-report vs progress) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (reserved; unlikely in this single-page layout) |

Exceptions:
- **Interactive hit targets (Back / Next / Install / Cancel, Browse / Resolve / Download / Deploy / Uninstall buttons): minimum 32px tall** — desktop pointer target, consistent with the Phase-1/2/3 button rule. This is a desktop app, not touch; a 44px touch target is not required.
- **FOMOD option row: minimum 40px tall** so the radio/checkbox control, the option **label**, and the option's required/recommended/notusable **type-state tag** fit on one comfortable line (the option *image*, when present, sits above or beside the row and is not bound by this height). This mirrors the Phase-3 40px download-row density rule — an explicit multiple-of-4 value above `md`.
- **FOMOD option image: max 96px tall preview** in the option list (a bounded thumbnail so a tall author image can't blow out the step), with the full-size image shown on hover/focus or inline at its natural width capped to the step body width. Missing image collapses to text with no reserved gap.
- **Per-mod Collection download row: minimum 40px tall** (same density as the Phase-3 download row — name, bar, percent/bytes, state share one line).
- **Progress bar height: 8px (`sm`)** — the slim `div` bar reused verbatim from Phase 3 for both per-mod and overall Collection progress.

## Typography

Inherited unchanged from the Phase-1/2/3 contract: 16px base, exactly 4 roles, exactly 2 weights (400 regular, 600 semibold), body line-height 1.5, headings 1.2. The new surfaces introduce no new sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (page `h1`, surface section `h2` e.g. "FOMOD installer", "Collections") | 24px | 600 | 1.2 |
| Heading (sub-section `h3`/`h4`, wizard step title, panel titles e.g. "Resolve report", "Conflict preview", current step name) | 18px | 600 | 1.2 |
| Body (option labels, option descriptions, mod names, button labels, resolve-report mod titles) | 16px | 400 (600 for emphasis — selected option label, step name, mod title) | 1.5 |
| Label / meta (option type-state tag Required/Recommended/Optional/NotUsable, step "N of M" counter, download percent + byte counts, mod-status tag Archived/Unavailable/Off-Nexus, Premium tag, error/conflict detail) | 14px | 400 (600 only for emphasis like a status tag) | 1.5 |

Monospace: staged file/archive names, install destinations in the dry-run conflict preview, plugin names, mod file names, and any path render in the Phase-1 `code` treatment (light-gray `#f3f3f3` background, 3px radius, `word-break: break-all`). Monospace is a treatment of the Body/Label roles, not a fifth size.

- The **primary "Download Collection"** CTA on the resolve report (enabled only after the user has seen the report; the single accent button on that screen).
- The **primary "Deploy"** CTA for an installed Collection (the same accent Deploy treatment already used by the Phase-2 conflict/profile surfaces).
- The **filled portion of an in-progress download bar** (`#0a66c2` fill on a `#f3f3f3` track) for both per-mod and overall Collection progress (reused verbatim from Phase 3).

- All secondary controls (Back, Cancel, Browse, Resolve, "Re-install with new choices", Refresh, dismiss-notice) stay neutral (Phase-1 default button chrome), NOT accent. **Uninstall Collection** is Destructive red (it purges + drops the profile), gated by a confirm modal.

## Surface & Interaction Contracts

These are the load-bearing contracts for this phase. The dry-run-before-apply and resolve-before-download safety gates must be visually unambiguous; the executor treats these as the source of truth. All three surfaces reuse the existing `.overlay`/`.modal`, `.empty`, `.warn`, `.report`, `.bar-track`/`.bar-fill`, and `button.cta`/`button.destructive` primitives — no new visual vocabulary.

### A. FOMOD Install Wizard (FOMOD-01, FOMOD-02)

1. **Entry & framing** — installing a mod that contains `fomod/ModuleConfig.xml` opens the wizard in the existing **modal/overlay** pattern (one wizard at a time, dims the page behind). The modal header shows the mod name (Heading) and a muted **"Step N of M · {step name}"** counter. A non-FOMOD archive installs through the existing Phase-1 direct path unchanged.
2. **One install step per screen** — the wizard renders exactly one install step at a time with a **Back** (neutral, hidden/disabled on the first step) and a single **Accent Next** button; on the final step the accent button becomes **Install**. A neutral **Cancel** closes the wizard and writes nothing to staging.
3. **Option groups by type** — within a step, each option group renders by its FOMOD type:
   - `SelectExactlyOne` / `SelectAtMostOne` → a **radio** group (AtMostOne allows a "none" choice).
   - `SelectAtLeastOne` / `SelectAll` / `SelectAny` → a **checkbox** group, honoring min/max; `SelectAll` options are pre-checked and disabled; min-not-met blocks **Next** with an inline Warning ("Select at least N option(s).").
   - Each option shows its **image** (from the staged archive, bounded ≤96px preview, missing → graceful text-only) **+ description** (Body, muted when the option is unselected).
4. **Conditional type states** — each option carries a type-state **tag** (Label 14px) and visual treatment: **Required** (pre-selected, locked, neutral tag), **Recommended** (Warning-amber tag, selectable), **Optional** (no tag), **CouldBeUsable** (Warning-amber tag), **NotUsable** (greyed/`muted`, control disabled, "Not available with your current choices" tooltip). Type states are author-driven and may change live (see §A.5).
5. **Live conditional re-evaluation** — as the user sets/unset flags via their choices, step **visibility**, option **type states**, and the resolved **conditionalFileInstalls** re-evaluate live (synchronously in the UI from the `crates/fomod` resolver's exposed conditions). An option flipping to NotUsable is auto-deselected and greyed; a step whose visibility condition is now false is skipped in the Back/Next sequence and the "N of M" counter reflects only currently-visible steps. No flash of an invalid selection is committed.
6. **Dry-run conflict preview (HARD SAFETY GATE — FOMOD-02)** — pressing **Install** does NOT immediately write to staging. It first shows a **conflict-preview panel** (reusing the `.report` treatment) listing the resolved file-install plan: which staged source lands at which destination (monospace), and any conflict, classified:
   - **Success** "No conflicts — safe to install." → the **Install** button (Accent) is enabled to apply.
   - **Warning** resolvable-by-priority overwrites (this mod's later option overwrites its earlier option, or a known mod-rank winner) → listed amber; Install enabled, with a one-line note that priority decides the winner.
   - **Error/Destructive** a **blocking** conflict (no determinable winner / incompatible required installs) → listed red; **Install is disabled** until the user changes a choice. The wizard never silently mis-installs (CONTEXT: "fail with a clear, specific error rather than silently mis-installing").
7. **Apply** — on a confirmed (non-blocking) Install, files route through the validated extract→staging path; the round-trip-pristine guarantee is untouched. On success the modal closes and the staged mod appears in the existing Phase-1/2 mod list as an ordinary `ManagedMod`.
8. **Malformed FOMOD** — a genuinely malformed/unsupported `ModuleConfig.xml` does not open a half-broken wizard; it surfaces the Phase-1 `run()` Error convention with the verbatim, specific parser reason ("Couldn't read this mod's FOMOD installer: {reason}.") and offers the neutral fallback to install the archive as a plain (non-guided) mod.
9. **Re-install to change choices** — re-running the FOMOD installer on an already-staged mod **re-stages** with fresh choices (replace the staged version) via a neutral **"Re-install with new choices"** affordance; editing choices in place is explicitly deferred (CONTEXT deferred-ideas).

### B. Collections — Browse, Resolve & Download (COLL-01, COLL-02)

1. **Premium gate (USER OVERRIDE — Premium-only v1)** — the Collections surface checks the logged-in account tier. A **free (non-Premium)** user sees a single, factual **Warning-styled notice** in place of the browse/download controls: heading "Collections require a NexusMods Premium account" + muted body explaining bulk in-app Collection download needs Premium (single-mod free-user `nxm://` from Phase 3 is unaffected). No per-mod free fallback is offered this phase. Not a blocking modal, not a repeated nag.
2. **Browse & select (COLL-01)** — a Premium user can enter/select a Collection for the currently-selected managed game (a Collection slug/URL input + the chosen revision). The surface only offers Collections for the selected game's domain (the Phase-3 `appid_for_domain` map gates mismatches with an inline notice "This Collection is for {game}, not your selected game.").
3. **Resolve report (HARD SAFETY GATE — COLL-02, success criterion #2)** — selecting a Collection first parses the revision manifest and shows a **resolve report BEFORE any download or disk write** (reusing the `.report` + list treatment). Each pinned mod is one row: mod **name** (Body), pinned **version** (monospace Label), and a **status tag**:
   - **Available** → Success `✓`, will be downloaded.
   - **Archived** → Warning amber tag — resolvable but flagged; included unless the user excludes it.
   - **Unavailable / removed** → Error red tag — cannot be fetched.
   - **Off-Nexus / externally-hosted** (script extenders etc.) → Warning amber tag "Manual step required" — **never auto-fetched**, surfaced as a listed manual instruction.
   A summary line counts each bucket ("X available · Y archived · Z unavailable · W manual"). **No partial disk mutation occurs before the user accepts the report.**
4. **Accept & download (COLL-02)** — the report carries a single **Accent "Download Collection"** CTA (enabled after the report renders) that proceeds with the **available** set; a neutral note states the unavailable/manual mods are skipped and recorded for the user. Bulk download reuses the Phase-3 client + the single shared `governor` rate limiter with bounded concurrency:
   - **Per-mod rows** — each downloading mod is a Phase-3-style download row (name, 8px Accent bar, percent + bytes, per-mod state: queued / downloading / extracting / done / failed / paused).
   - **Overall progress** — one aggregate bar + "M of N mods" caption above the per-mod list.
   - **Rate-limit notice** — the same single Warning inline notice from Phase 3 ("Pausing to respect NexusMods rate limits — downloads will resume automatically.") when the client backs off; never a freeze, never a blocking modal.
   - A failed per-mod row shows the verbatim reason (Error) + a neutral **"Retry"**; one failed mod does not abort the batch.
5. **Empty / no-selection state** — before a Collection is chosen: a muted empty block, heading "No Collection selected" + body "Paste a NexusMods Collection link for {game} to see what it installs before anything downloads."

### C. Installed Collection — Apply, Deploy & Uninstall (COLL-03, COLL-04, COLL-05)

1. **Collection = a profile** — an installed Collection appears as a dedicated, isolated **Phase-2 profile** in the existing profile selector (e.g. named after the Collection), reusing the active-dot/active-label/selector chrome verbatim. Its FOMOD choices were replayed headlessly from the manifest during install (no interactive wizard per mod) and its rules are already mapped onto the existing conflict/priority + plugin load-order model — there is **no new parallel rules surface**; the user sees the Collection's mods/plugins through the existing Conflict and Plugin views.
2. **Manual-steps panel (COLL-03)** — if the resolve report flagged off-Nexus/manual mods, the installed Collection shows a persistent **Warning-styled "Manual steps remaining"** panel listing each required manual action (e.g. "Install {script extender} separately"), so the user knows the Collection isn't fully automatic. Dismissable per-Collection, not auto-cleared.
3. **Deploy (COLL-04)** — deploying an installed Collection is the existing Phase-2 profile-switch path: a single **Accent "Deploy"** activates the Collection's profile → deploy-winners → apply-load-order, surfacing the existing `SwitchReport`/`DeployReport` `.report`. No new deploy primitive, no new visual treatment.
4. **Uninstall (COLL-05 — destructive, fully reversible)** — a neutral **"Uninstall Collection"** affordance opens a confirm modal with a **Destructive red** confirm. Copy states exactly what happens: purge-to-pristine via the deploy engine + drop the Collection's profile + remove its staged mods, leaving the game byte-for-byte vanilla. On success the existing purge/verify `.report` shows the pristine result (Success when pristine, Error if drift). This reuses the Phase-1/2 reversibility guarantee and is the hard UAT for the phase.

Destructive-action affordance rule (carried from Phases 2–3): every disk-/credential-mutating action is explicit and gated. **Uninstall Collection** mutates the game folder + drops a profile + deletes staged files, so it uses the **Destructive red** confirm in a modal. **Deploy** is non-destructive (reversible by design) and stays Accent, no extra confirm beyond the existing profile-switch flow. The FOMOD **dry-run preview** and the Collection **resolve report** are the two non-skippable "see consequence before acting" gates — neither writes to disk until the user presses the accent CTA that follows it. Cancelling a wizard or a per-mod download is a single neutral "Cancel" with no modal (non-destructive — nothing was staged).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| untrusted archive → fomod parser | A downloaded/local mod archive's `fomod/ModuleConfig.xml` is attacker-authorable untrusted XML |
| FOMOD source/destination → staging tree | `<file>/<folder>` `source`/`destination` strings cross into filesystem path resolution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | FOMOD `source`/`destination` path traversal (`../`, absolute) | mitigate | resolve `source` only inside the validated staged tree; `destination` is applied later through `extract::install_archive` + the existing `conflict::resolve` path-escape guard (Plan 02) — `resolve` here emits relative paths only and never writes; destination must stay under `Data/` |
| T-04-02 | Denial of Service | XML entity-expansion / billion-laughs in ModuleConfig.xml | mitigate | quick-xml does not expand external entities by default (no DTD processing); the file is config-sized (KB); parse maps any failure to a specific `FomodError::Xml`, never hangs |
| T-04-03 | Tampering | Malformed/unsupported XML silently mis-installing | mitigate | every unsupported construct returns `FomodError::MalformedSchema`/`Xml` (the locked "fail clearly, never mis-install"); resolve is pure and surfaces the plan before any write |
| T-04-04 | Tampering | Wrapper-folder mis-detection corrupting source resolution | mitigate | `detect_archive_root` uses an explicit recognized-root token list; unit-tested against wrapper/no-wrapper/nested fixtures so a real multi-folder mod is never wrongly flattened |
</threat_model>

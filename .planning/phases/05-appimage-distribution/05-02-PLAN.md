<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| crates.io / GitHub Actions -> shipped binary | Transitive crate deps and the build action determine what native code lands in the AppImage. The supply-chain gate and bundled-binary review are the controls. |
| Build host (CI runner) -> distributed AppImage | The artifact uploaded to the GitHub Release is what users run; its bundled `.so` set must contain no non-free / no fragile system-OpenSSL code. |
| GitHub Release (HTTPS) -> user | The v1 distribution channel. No code-signing/provenance in v1 (deferred). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04 | Information disclosure / legal | shipped binary contents (UnRAR) | mitigate | `unrar`/`unrar_sys` banned in `deny.toml` (per-push gate, re-run in `release.yml`); `.rar` shells out to a system tool; `scripts/dist-audit.sh` proves UnRAR-absence in the built AppImage and `DIST-AUDIT.md` records it by name. |
| T-05-05 | Tampering | vulnerable/yanked transitive dep in the shipped binary | mitigate | `cargo deny check advisories bans licenses sources` re-run in `release.yml` fails on a yanked/vulnerable/banned dep; the result is captured in `DIST-AUDIT.md`. |
| T-05-06 | Tampering | TLS path (dynamic system OpenSSL would break the self-contained/rustls guarantee) | mitigate | `reqwest` is rustls-only (project convention); the `ldd` step in `scripts/dist-audit.sh` confirms no app-path `libssl`/`libcrypto`, recorded in `DIST-AUDIT.md` (V6). |
| T-05-07 | Spoofing | unsigned release artifact (no provenance) | accept | Code-signing/notarization is explicitly deferred to v2 (CONTEXT Deferred Ideas). v1 ships over the HTTPS GitHub Release channel; documented as an accepted v1 limitation. |
| T-05-SC | Tampering | `tauri-apps/tauri-action` GitHub Action | mitigate | First-party Tauri-org action (RESEARCH Package Legitimacy Audit: verdict OK); pin to a concrete release ref (`action-v0.6.2`) rather than a floating branch for reproducibility. No new crates.io packages are installed this phase, so no per-package legitimacy checkpoint is required. |
</threat_model>

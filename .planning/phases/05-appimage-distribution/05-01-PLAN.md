<objective>
Deliver the first vertical slice of DIST-01: a NexTwist build that bundles cleanly into a
launchable AppImage and self-tests its own `nxm://` handler registration on startup.

Two real blockers stand between the current tree and a buildable AppImage: the desktop icon
is 32x32 (linuxdeploy expects >=128x128), and the success criterion "self-test passes" has no
call site. This plan fixes the icon set, wires the plugin's own `is_registered("nxm")`
self-test into the existing startup hook (non-fatal, per the locked decision), points the
bundle config at the larger icon, and adds a headless test proving the wiring compiles and
never aborts startup.

Purpose: After this plan a developer can run `cargo tauri build --bundles appimage` without
the icon blocker, and a launched build logs a `nxm://` handler self-test result — the
end-to-end "buildable + self-testing AppImage" capability that Plan 02's release pipeline and
audit then package and verify.

Output: regenerated >=128x128 icon set, `bundle.icon` updated, `is_registered("nxm")`
self-test in `lib.rs`, and `src-tauri/tests/nxm_self_test.rs`.
</objective>

## Phase Goal

**As a** Linux gamer (or distro packager), **I want to** download and run NexTwist as a
single-file AppImage that registers its own `nxm://` handler on first run, **so that** the
"Mod Manager Download" one-click flow works with no install friction and the app can prove its
handler is wired correctly.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS desktop session -> NexTwist shell | The `nxm://` MIME default + the written `nextwist-handler.desktop` `Exec=` are OS-registered state the shell writes via the deep-link plugin. A stale/ephemeral `Exec=` is a hijack surface if it points at a deleted `/tmp/.mount_*` path. |
| Build host -> shipped binary | Icons and bundle config become part of the distributed AppImage; nothing here links new native code. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | `nextwist-handler.desktop` `Exec=` path | mitigate | Use the plugin's `register_all()` (reads `$APPIMAGE` for a durable absolute path) — unchanged; this plan adds only the `is_registered("nxm")` self-test that surfaces a WARN when NexTwist is not the registered default, making a stale/hijacked handler observable. Durable-`Exec=` confirmation is a manual UAT item (real AppImage launch). |
| T-05-02 | Denial of service | startup `setup` hook | mitigate | The self-test is strictly non-fatal (no `?`/`unwrap`/`expect`); a missing `xdg-mime` on a minimal distro logs a WARN and the app still opens (locked decision: warn-and-continue). Asserted by `tests/nxm_self_test.rs`. |
| T-05-03 | Information disclosure | self-test logging | accept | The self-test logs only PASS/WARN and the error cause — it never logs an incoming `nxm://` URL (the existing `on_open_url` already avoids logging URLs that may carry key/expires/code, V7). No new disclosure surface. |
</threat_model>

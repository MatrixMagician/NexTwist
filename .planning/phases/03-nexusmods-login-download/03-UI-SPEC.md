# Phase 3 — UI Design Contract

> Visual and interaction contract for Phase 3 (NexusMods Login & Download). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> This phase extends the existing "functional-minimal Svelte 5" UI (`frontend/src/routes/+page.svelte`) established in Phases 1–2. It adds two new surfaces — a minimal **account/login panel** and a **downloads list with per-item progress** — plus an `nxm://` deep-link toast. No component library is added; all visual tokens below are inherited verbatim from the Phase-1/Phase-2 stylesheet so the new surfaces are visually continuous with what already ships.

**Design priority for this phase:** trust and legibility of state. Auth state (logged-in vs logged-out, Premium vs free) and download progress must be unambiguous at a glance, and the no-plaintext-token security invariant (NEXUS-02) must be visible to the user when a keyring backend is missing. Downloads must never freeze the UI (async progress events drive every bar).

## Spacing Scale

Inherited unchanged from the Phase-2 contract (4px grid, `1rem = 16px`). The new account panel and downloads list reuse the existing section/list/row spacing so they sit flush with the Phase-1/2 surfaces.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps, glyph-to-text gap, progress-bar-to-label gap, dense row vertical padding |
| sm | 8px | Compact list-row gaps, status-dot-to-text gap, button vertical padding, account-panel inline gaps |
| md | 16px | Default element spacing, section inner padding, gap between account fields |
| lg | 24px | Section padding, gap between the account panel and downloads list, modal padding |
| xl | 32px | Layout gaps between major blocks |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (reserved; unlikely in this single-page layout) |

Exceptions:
- **Interactive hit targets (Login / Logout / Cancel-download buttons): minimum 32px tall** (desktop pointer target; this is a desktop app, not touch — 44px touch target not required), consistent with the Phase-2 reorder/toggle rule.
- **Download-row height: minimum 40px** so the per-item progress bar, name (monospace, may wrap/ellipsize), percent, and cancel control fit on one comfortable line. This is a list-density choice, an explicit multiple-of-4 value above `md`.
- **Progress bar height: 8px (`sm`)** — a slim native-or-div bar; this is the bar's own thickness, not content padding.

## Typography

Inherited unchanged from the Phase-2 contract: 16px base, exactly 4 roles, exactly 2 weights (400 regular, 600 semibold), body line-height 1.5, headings 1.2. The new surfaces introduce no new sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (page `h1`, surface section `h2`) | 24px | 600 | 1.2 |
| Heading (sub-section `h3`/`h4`, panel titles e.g. "Account", "Downloads") | 18px | 600 | 1.2 |
| Body (list rows, descriptions, button labels, logged-in username) | 16px | 400 (600 for the username/emphasis) | 1.5 |
| Label / meta (download percent, byte counts, rate-limit caption, "Premium"/"Free" tag, error detail) | 14px | 400 (600 only for emphasis like an account-tier tag) | 1.5 |

Monospace: downloaded file/archive names, `nxm://` URLs, and staging paths render in the Phase-1 `code` treatment (light-gray `#f3f3f3` background, 3px radius, `word-break: break-all`). Monospace is a treatment of the Body/Label roles, not a fifth size.

## Color

Light theme continues from Phases 1–2 (no dark mode in scope). The 60/30/10 split and the amber/green/red/muted semantics are inherited verbatim so warnings and status read identically across every phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Page/main background, account panel and downloads list area |
| Secondary (30%) | `#f3f3f3` (surfaces) with `#ccc` borders, section radius 8px | Section/card chrome, download-row container, `code`/URL chips, progress-bar track (unfilled portion), modal panel |
| Accent (10%) | `#0a66c2` (calm blue) | Reserved — see list below |
| Destructive | `#cf222e` (Phase-1 `.err` red) | Logout confirm emphasis + hard token-storage failure message + cancel-download affordance |

Accent reserved for (explicit — never "all interactive elements"):
- The single **primary Login CTA** in the logged-out account panel ("Log in with NexusMods").
- The **logged-in status dot** (`●`) next to the connected account name.
- The **filled portion of an in-progress download bar** (`#0a66c2` fill on a `#f3f3f3` track).
- Secondary buttons (Logout, "Use API key instead", Cancel download, dismiss-toast) stay neutral (Phase-1 default button chrome), NOT accent. Logout opens a neutral confirm; only the destructive token-storage-failure message uses red text.

Additional semantic colors (carried verbatim from Phases 1–2 — these are status, not part of the 10% accent budget):

| Semantic | Value | Usage |
|----------|-------|-------|
| Success | `#1a7f37` | "Logged in", a completed download (`✓ Done`), "Added to staging — ready to deploy" |
| Warning | text `#9a6700`, bg `#fff8e5`, border `#e6c200` | Rate-limit throttle notice ("Pausing to respect NexusMods limits…"), expired/invalid `nxm://` link, free-user "open the website to download" hint |
| Error/Destructive | `#cf222e` | A failed download, the **no-keyring-backend hard-fail** banner (NEXUS-02), OAuth/redeem failures |
| Muted | `#777` | Logged-out helper text, empty-downloads text, completed-but-idle byte counts, secondary captions |

## Surface & Interaction Contracts

These are the load-bearing contracts for this phase. The auth-state and download-progress semantics must be visually unambiguous; the executor treats these as the source of truth.

### A. Account / Login Panel (NEXUS-01, NEXUS-02)

1. **Logged-out state** — a compact panel with a heading "Account", muted helper text "Log in to download mods from NexusMods.", and a single **Accent primary button "Log in with NexusMods"** (launches OAuth2 + PKCE via the `nxm://oauth/callback` redirect). Below it, a neutral secondary affordance "Use an API key instead" reveals an inline key-paste field + "Save key" button (the last-resort v1 fallback; D-Auth).
2. **Logged-in state** — the same panel collapses to: a green Success status dot `●`, the connected account **username** (Body 600), an account-tier **tag** ("Premium" or "Free", Label 14px, neutral chip in the Secondary `#f3f3f3` style), and a neutral **"Log out"** button. No token or key is ever displayed.
3. **In-flight auth** — while the OAuth round-trip is pending, show the Phase-1 `busy` treatment ("Connecting to NexusMods…", italic muted) and disable the Login button; surface OAuth failure with the Phase-1 `run()` error convention.
4. **No-keyring hard-fail (NEXUS-02, critical security invariant)** — if no Secret Service / keyring backend is available, login is **blocked** with an Error/Destructive-styled banner: heading "Can't store your login securely" + body explaining NexTwist refuses to save credentials as plaintext and how to enable a keyring. The login button is disabled in this state. NexTwist NEVER falls back to a plaintext file — the banner replaces login, it does not offer a downgrade.

### B. Downloads List (NEXUS-03, NEXUS-05, NEXUS-06)

1. **Downloads list** — a vertical list of download rows under a "Downloads" heading. Each row: the mod/file **name** (monospace Body, ellipsized/`break-all`), a **per-item progress bar** (8px, Accent fill on Secondary track), a **percent + transferred/total byte count** (Label 14px, muted), and a per-item **state** affordance. Rows are driven entirely by async progress events so the UI never freezes (success-criterion #4).
2. **Per-item states** — each row reflects exactly one state, visually distinct:
   - *Queued* — muted, bar at 0%, caption "Queued".
   - *Downloading* — Accent-filled bar advancing, percent + bytes, a neutral **"Cancel"** affordance.
   - *Extracting* — bar full + indeterminate/striped or caption "Extracting…" (reuses the Phase-1 `extract`→staging pipeline; NEXUS-06).
   - *Done* — Success `✓ Done` + "Added to staging — ready to deploy" (links the user back to the existing deploy flow). The staged mod then appears in the Phase-1/2 mod list as an ordinary `ManagedMod`.
   - *Failed* — Error-styled row with the verbatim backend reason + a neutral "Retry" affordance.
3. **Concurrency + rate-limit notice (NEXUS-05)** — v1 caps concurrent downloads (small cap, no queue UI beyond the list). When the client backs off to honor `X-RL-*` headers, show a single Warning-styled inline notice above the list: "Pausing to respect NexusMods rate limits — downloads will resume automatically." Never a blocking modal; never a UI freeze.
4. **Empty state** — when no downloads exist: a muted empty block, heading "No downloads yet" + body "Use the Log in panel, then start a download from NexusMods. Free accounts: use the website 'Mod Manager Download' button."

### C. nxm:// One-Click Handoff (NEXUS-04, NXM-01)

1. **Deep-link arrival** — when the app receives an `nxm://` link (from the website "Mod Manager Download" button, routed via the deep-link + single-instance plugins), it appears as a **new row in the Downloads list** (Section B) rather than a separate surface; a brief, non-blocking **toast/inline notice** confirms "Download started from NexusMods" (Success, auto-dismiss). A second invocation while the app is open is forwarded to the live instance and produces one new row — never a duplicate window.
2. **Free-user guidance** — the logged-in "Free" account tier shows a one-line muted hint near the Downloads heading: "Free account: start downloads from the NexusMods website 'Mod Manager Download' button." Premium users get a hint that in-app direct download is available. No paywall nag beyond this factual hint.
3. **Invalid / expired link** — a keyed `nxm://` link that has expired or fails to redeem surfaces a Warning-styled inline notice: "This download link has expired. Re-open it from the NexusMods website." — it does NOT create a stuck/failed row that looks like a download error.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA — login (logged out) | "Log in with NexusMods" |
| Secondary CTA — key fallback | "Use an API key instead" → reveals field + "Save key" |
| Primary CTA — logout (logged in) | "Log out" |
| In-flight auth | "Connecting to NexusMods…" |
| Logged-in status | "{username}" + tier tag "Premium" / "Free" |
| Empty state — downloads | Heading: "No downloads yet" · Body: "Use the Log in panel, then start a download from NexusMods. Free accounts: use the website 'Mod Manager Download' button." |
| Download — downloading | Caption: "{pct}% · {transferred} / {total}" + "Cancel" |
| Download — extracting | "Extracting…" |
| Download — done | "✓ Done — added to staging, ready to deploy" |
| nxm:// arrival toast | "Download started from NexusMods" |
| Free-user hint | "Free account: start downloads from the NexusMods website 'Mod Manager Download' button." |
| Rate-limit notice | "Pausing to respect NexusMods rate limits — downloads will resume automatically." |
| Error — no keyring (NEXUS-02) | Heading: "Can't store your login securely" · Body: "NexTwist won't save your NexusMods credentials as plaintext. Enable a system keyring (GNOME Keyring or KWallet) and try again." |
| Error — OAuth/redeem failure | "Login failed: {reason}. Try again, or use an API key instead." |
| Error — download failed | "Download failed: {reason}." + "Retry" |
| Error — expired nxm:// link | "This download link has expired. Re-open it from the NexusMods website." |
| Error state (generic, Phase-1 pattern) | "{action} failed: {reason}" — keep the Phase-1 `run()` `label + error` convention; surface the backend error string verbatim so auth/security failures are never hidden. |
| Destructive confirmation — Log out | "Log out of NexusMods?": "This clears your saved login from the system keyring. You'll need to log in again to download mods." — Confirm: "Log out", Cancel: "Cancel". |

Destructive-action affordance rule (carried from Phase 2): every disk-/credential-mutating action is explicit. **Log out** clears the keyring entry and in-memory token (D-Auth) and is confirmation-gated with a neutral confirm (it removes a credential, not user data, so it does not require Destructive red — but the no-keyring hard-fail message and failed-download rows DO use the Destructive/Error red). Cancelling an in-flight download is a single neutral "Cancel" with no modal (it is non-destructive — the staged result simply isn't produced).

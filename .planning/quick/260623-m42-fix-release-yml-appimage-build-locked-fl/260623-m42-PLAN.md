# Quick Task 260623-m42 — Fix release pipeline: AppImage build + version + changelog

## Problem (diagnosed)

The `v1.0` tag triggered `release.yml`, which **failed** at the bundle step (run 28033312651):

```
running tauri [ 'build', '--bundles', 'appimage', '--locked' ]
error: unexpected argument '--locked' found
  tip: to pass '--locked' as a value, use '-- --locked'
```

`tauri build` does not accept `--locked` directly — it must come after `--` to be forwarded
to the underlying `cargo build`. So no GitHub Release / AppImage was ever produced. Two
secondary gaps: the app version is still `0.1.0` (tag is `v1.0`), and the release has no
changelog body.

The release `[profile.release]` is already optimized (`lto = "thin"`, `strip = true`) — no
profile change needed; "optimized AppImage" means the release-profile build the workflow
produces.

## Tasks

### Task 1 — Fix the `--locked` flag + wire a changelog into the release body
**Files:** `.github/workflows/release.yml`
- Change the tauri-action `args` from `--bundles appimage --locked` to `--bundles appimage -- --locked`.
- Add a step before tauri-action that extracts the top-most `## [version]` section from
  `CHANGELOG.md` into a multi-line `$GITHUB_OUTPUT` (`notes`).
- Add `releaseBody: ${{ steps.changelog.outputs.notes }}` to the tauri-action `with:` block.
- **verify:** `grep -- '-- --locked' release.yml`; the changelog step + releaseBody present.
- **done:** the only flag change is the `--` insertion; gate steps (test/clippy/deny) untouched.

### Task 2 — Bump product version to 1.0.0
**Files:** `src-tauri/tauri.conf.json`
- Set `version` from `0.1.0` to `1.0.0`.
- **verify:** `node -e "console.log(require('./src-tauri/tauri.conf.json').version)"` → `1.0.0`.
- **done:** the AppImage filename/version and the release will show 1.0.0.

### Task 3 — Create CHANGELOG.md
**Files:** `CHANGELOG.md`
- Keep a Changelog format with a `## [1.0.0] - 2026-06-23` section summarizing the v1.0 MVP
  (sourced from `.planning/MILESTONES.md`), plus an `Unreleased` heading and a Known Limitations note.
- **verify:** top-most section is `## [1.0.0]`; the release.yml extraction picks it up.
- **done:** durable changelog the release body reads from.

## Out of scope / follow-up (needs user sign-off)
Re-triggering the release requires moving the published `v1.0` tag to the fixed commit
(force-update). The original release produced no artifact, so re-pointing is safe — but it is
a release action, surfaced to the user after the config fix lands. Future releases should tag
`vX.Y.Z` (full semver) to match the changelog/version.

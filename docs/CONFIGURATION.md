# Configuration

NexTwist is a desktop application, not a service. It has **no user-facing
configuration file**: there is no `.env`, no `config.json`, and the
`tauri-plugin-store` settings store is not used. Almost everything NexTwist
needs is either discovered at runtime (Steam install, Proton prefix) or held in
internal state (the SQLite database, the OS keyring).

This document describes the few things that *are* configurable or that influence
runtime behaviour: a small set of environment variables the engine honours, the
on-disk state locations, and the build-time policy/config files that pin the
project's toolchain and dependencies.

## Environment Variables

NexTwist does not require any environment variables to run. It **reads** a small
number of standard variables when they are present; none are NexTwist-specific
and none are mandatory.

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `STEAM_COMPAT_DATA_PATH` | Optional | Derived from the Steam library layout | Per-game Proton compatdata root. When set, NexTwist uses `$STEAM_COMPAT_DATA_PATH/pfx` as the Wine/Proton prefix instead of deriving `<library>/steamapps/compatdata/<appid>/pfx`. Honoured in `crates/steam/src/resolve.rs`. Steam exports this when launching a game, so it is mainly relevant if NexTwist is run from inside a game's launch environment. |
| `HOME` | Optional | ŌĆö | Used to locate the default Steam install directory (`crates/steam/src/discover.rs`). On a normal Linux desktop this is always set; if absent, Steam auto-discovery is skipped and the user can add a game by folder instead. |
| `PATH` | Optional | ŌĆö | Searched for the optional `unrar` and `7z` binaries used to extract `.rar` mod archives (`crates/extract/src/rar.rs`), and for the `xdg-mime` / `update-desktop-database` tools used to register the `nxm://` handler. RAR support is degraded gracefully if neither binary is found. |
| `APPIMAGE` | Optional (set by the AppImage runtime) | ŌĆö | Read by the deep-link plugin's `register_all()` to write a durable absolute `Exec=` path into the `nxm://` desktop handler when running as a shipped AppImage. Set automatically by the AppImage runtime; you do not set this yourself. <!-- VERIFY: $APPIMAGE is exported by the AppImage runtime at launch; not directly read in NexTwist source ŌĆö consumed inside tauri-plugin-deep-link -->|

There are no required environment variables that cause startup to fail if
missing. NexTwist starts even on a minimal system; missing optional tools only
degrade the corresponding feature (e.g. no RAR extraction, or the `nxm://`
handler not registering as default).

## Config File Format

NexTwist has no application configuration file. All persistent data lives in two
places, both created and managed by the app itself:

### SQLite database (`nextwist.db`)

The single source of truth for the game registry, the per-file deploy manifest,
the operation journal, the vanilla backup ledger, and the mod/profile/plugin
tables. It is opened at startup by `AppState::init` (`src-tauri/src/state.rs`):

- **Filename:** `nextwist.db`
- **Location:** inside the OS application-data directory NexTwist owns. This is
  resolved via Tauri's `app.path().app_data_dir()` and the file is placed at
  `<app-data-dir>/nextwist.db`. If the app-data dir cannot be resolved, NexTwist
  falls back to a relative `.nextwist` directory (`resolve_data_dir` in
  `src-tauri/src/lib.rs`).
  <!-- VERIFY: concrete runtime path of app_data_dir for identifier "com.nextwist.app" on a given Linux desktop (typically ~/.local/share/com.nextwist.app) ŌĆö resolved by Tauri at runtime, not hard-coded in the repo -->
- **Schema:** versioned, additive migrations in `crates/store/src/migrations/V*.sql`,
  applied automatically on open. The database file is created if it does not exist.

You do not edit this file by hand. The hard invariant (`crates/store`) is that no
`rusqlite` type appears in the store's public API ŌĆö all SQL stays inside the
`store` crate.

### NexusMods credential (OS keyring)

The long-lived NexusMods credential (OAuth refresh token or API key) is **not**
stored in any file. It is written to the OS Secret Service via the `keyring`
crate (`src-tauri/src/keyring.rs`):

- **Service name:** `nextwist`
- **Account / key:** `nexusmods-refresh-token`
- **Backend:** the Linux Secret Service (GNOME Keyring / KWallet) via
  `keyring` 3.6 with the `sync-secret-service` + `crypto-rust` features.

Hard invariant (NEXUS-02): if no Secret Service backend is available, store/load
operations return `NoKeyringBackend` and **nothing is written to disk**. There is
deliberately no plaintext fallback ŌĆö login is blocked rather than persisting the
credential insecurely.

## Required vs Optional Settings

There are no settings that must be supplied for NexTwist to start. The
application is designed to launch on a minimal system and surface missing
capabilities through the UI rather than failing at startup.

**Hard runtime requirement (not a "setting"):**

- A WebKitGTK 4.1 runtime must be present for the desktop window to render. This
  is a system dependency of the Tauri shell, documented at build time in
  `Cargo.toml`. <!-- VERIFY: the exact minimum WebKitGTK runtime version required on an end-user system ŌĆö runtime dependency, not pinned in the repo -->

**Optional capabilities (each degrades gracefully if absent):**

- **OS Secret Service** ŌĆö required only for NexusMods login. Absent ŌåÆ login is
  disabled (NEXUS-02 hard-fail banner), but local mod management still works.
- **`unrar` or `7z` on `PATH`** ŌĆö required only to extract `.rar` archives.
  Absent ŌåÆ `.rar` mods cannot be installed; `.zip` / `.7z` (handled in-process)
  are unaffected.
- **`xdg-mime` + `update-desktop-database` on `PATH`** ŌĆö required only to
  register NexTwist as the default `nxm://` handler. Absent ŌåÆ one-click Nexus
  downloads will not route to NexTwist, but the app still opens (the startup
  self-test logs a WARN; see `nxm_self_test` in `src-tauri/src/lib.rs`).

## Defaults

NexTwist's runtime defaults are derived from the system rather than from a
defaults file:

| Setting | Default | Where it comes from |
|---------|---------|---------------------|
| Application data directory | OS app-data dir for identifier `com.nextwist.app`; falls back to `.nextwist` if unresolved | `resolve_data_dir` (`src-tauri/src/lib.rs`) |
| Database file | `nextwist.db` inside the app-data directory | `AppState::init` (`src-tauri/src/state.rs`) |
| Proton prefix | `<steam-library>/steamapps/compatdata/<appid>/pfx`, unless `$STEAM_COMPAT_DATA_PATH` overrides it | `proton_prefix` (`crates/steam/src/resolve.rs`) |
| Keyring service / account | `nextwist` / `nexusmods-refresh-token` | `SERVICE` / `USER` constants (`src-tauri/src/keyring.rs`) |
| `nxm://` deep-link scheme | `nxm` | `tauri.conf.json` ŌåÆ `plugins.deep-link.desktop.schemes` |
| Main window size | 960 x 720, resizable | `tauri.conf.json` ŌåÆ `app.windows` |
| App identifier | `com.nextwist.app` | `tauri.conf.json` ŌåÆ `identifier` |

## Per-Environment Overrides

NexTwist does not support development / staging / production configuration
profiles ŌĆö it is a single-mode desktop app. The only environment-driven
behaviour comes from the standard variables above (chiefly
`STEAM_COMPAT_DATA_PATH`) and from whether the app is running from source versus
as a packaged AppImage:

- **Run from source (`cargo tauri dev`):** the `nxm://` handler is registered
  against the dev binary using `xdg-mime` / `update-desktop-database`. The
  frontend dev server runs at `http://localhost:5173` (`tauri.conf.json` ŌåÆ
  `build.devUrl`).
- **Packaged AppImage:** the deep-link plugin reads `$APPIMAGE` so the
  registered `nxm://` handler points at a durable absolute path to the AppImage
  rather than a transient mount.

There is no separate configuration step to switch between these; the app detects
its environment at runtime.

## Build-Time Configuration

These files configure how NexTwist is *built* rather than how it *runs*. They
are policy/toolchain files, not user settings, but they are the authoritative
place to change pinned versions and supply-chain rules.

| File | Purpose |
|------|---------|
| `rust-toolchain.toml` | Pins the Rust toolchain to the `stable` channel with `rustfmt` + `clippy`. MSRV is `1.89` (required by `libloot`; subsumes the `1.85` floor that EXDEV deploy detection needs). |
| `Cargo.toml` (workspace root) | Single source of pinned dependency versions in `[workspace.dependencies]`; member crates reference them via `<dep>.workspace = true`. Notable pins: `rusqlite 0.39` (bundled, capped by `refinery 0.9.2`), `reqwest 0.13` with `rustls` only (no native-tls/OpenSSL), `tauri 2.11`. Release profile uses `lto = "thin"` and `strip = true`. |
| `deny.toml` | `cargo-deny` policy. Bans the non-free `unrar` / `unrar_sys` crates (RAR support shells out to a system binary instead), and gates allowed licenses to a permissive + GPL-compatible set. CI fails if a dependency trips these rules. |
| `src-tauri/tauri.conf.json` | Tauri shell config: product name, app identifier (`com.nextwist.app`), window definition, the `nxm` deep-link scheme, the embedded frontend path (`../frontend/build`), and AppImage bundle settings. |

To change a dependency version, edit `[workspace.dependencies]` in the root
`Cargo.toml` rather than individual crate manifests. To change the allowed
license set or the crate bans, edit `deny.toml` and re-run
`cargo deny check advisories bans licenses sources`.
